## 1. Existing font infrastructure (`src/fonts/`, `src/writer/font_manager.rs`)

Total `src/fonts/` size: **19,323 lines** across 13 files (data tables dominate — `adobe_glyph_list.rs` alone is 4,292 lines).

### What's built and tested

| Component | Location | Status |
|---|---|---|
| TrueType parser wrapping `ttf-parser` | `src/fonts/truetype_parser.rs` (1,080 lines) | ✅ Complete. Builds Unicode→GID, glyph widths, font flags. Generates widths-array string + ToUnicode CMap string. |
| `EmbeddedFont` struct (write-side) | `src/writer/font_manager.rs:840+` | ✅ Owns face bytes, parses metrics, exposes `glyph_id`, `glyph_width`, `text_width`, `use_string`, `encode_string` (Identity-H hex format), `generate_widths_array`, `generate_tounicode_cmap`. |
| `EmbeddedFontManager` registry | `src/writer/font_manager.rs:1090+` | ✅ Register by name, get by name, allocate resource IDs. |
| `FontSubsetter` glyph tracker | `src/fonts/font_subsetter.rs` (354 lines) | 🟡 Tracks used codepoints + glyph IDs; generates 6-letter subset tag. **Does not emit subset binary** — comment at lines 19–20 explicitly defers binary subsetting to "a future version or external crate." |
| ToUnicode CMap helpers | `src/fonts/cmap.rs`, `src/fonts/character_mapper.rs`, `src/fonts/encoding.rs` | ✅ Read-side parsers + write-side `UnicodeEncoder::encode_identity_h`. |
| 70 unit tests (`tests/test_unicode_font_embedding.rs`, `tests/test_cjk_script_support.rs`) | tests/ | ✅ All pass. Cover glyph tracking, encoding, CMap format, subset stats, CJK script detection. |
| `FontInfo` Base-14 path | `src/writer/font_manager.rs:200+` | ✅ Helvetica/Times/Courier/Symbol/ZapfDingbats families, weights, italics. |

### What's missing

1. **PDF object emission for embedded fonts.** `EmbeddedFont` exposes string-form helpers (CMap, widths array) but nothing serialises a Type 0 dict, a CIDFontType2 dict, a FontDescriptor dict, a FontFile2 stream, or a ToUnicode stream as actual PDF indirect objects. `src/fonts/font_dict.rs` (7,283 lines) is exclusively the *read* path — parsing existing PDF font dicts. There's no symmetric `FontDictBuilder`.
2. **Wiring into `PdfWriter`.** `grep EmbeddedFont src/writer/pdf_writer.rs src/writer/document_builder.rs src/api/pdf_builder.rs` returns zero hits. The type exists but no PDF generation path uses it. `Pdf::from_html()` today only sees Base-14 fonts.
3. **Real binary subsetter.** `FontSubsetter` records intent; the full font is currently embedded. For high-volume HTML→PDF this is a file-size problem, not a correctness one.
4. **System font discovery.** `fontdb = "0.23"` is declared optional under feature `rendering` (Cargo.toml:168) but unused on the writer side.
5. **Shaping.** `rustybuzz = "0.20"` is declared optional under feature `rendering` (Cargo.toml:169) but unused on the writer side. Without shaping, complex scripts (Arabic, Devanagari) cannot be rendered correctly even with the right font.
6. **End-to-end round-trip.** No test produces a real PDF with an embedded custom font and verifies that `extract_text` returns the input string.

### Revised FONT sub-tasks

Plan v1 had 6 sub-tasks (FONT-1..FONT-6). Audit reduces them to:

- ~~FONT-1 (ttf-parser + EmbeddedFont type)~~ — **already done.** Promote to "verify + smoke-test what exists."
- **FONT-2 — real binary subsetter via `subsetter` crate.** Replace `FontSubsetter`'s "track only" with `subset_font_bytes(face: &[u8], used_glyphs: &BTreeSet<u16>) -> Vec<u8>`. Wire into `EmbeddedFont`. Acceptance: a 100 KB font subsets to <30 KB for typical English content.
- **FONT-3 — PDF object emission + PdfWriter wiring.** New module `src/writer/font_pdf_objects.rs` with `EmbeddedFont::write_pdf_objects(writer: &mut PdfWriter) -> FontResources` returning the indirect refs. Emits five objects per font: Type0 dict, CIDFontType2 dict, FontDescriptor, FontFile2 stream (subsetted bytes), ToUnicode stream. PdfWriter resource dict picks them up.
- **FONT-4 — fontdb system discovery + family matching.** Move fontdb out of the `rendering` feature into a new `system-fonts` feature (default on for bindings, off for WASM). Walk OS font dirs lazily; cache the db on `PdfBuilder`. Match `font-family` list with weight + style + script-coverage filtering. Deterministic tie-break by face path.
- **FONT-5 — rustybuzz shaping.** Same feature treatment. Build `shape(text, font, size, script, direction)` returning `ShapedRun { glyphs: Vec<{glyph_id, x_advance, x_offset, cluster}> }`. Use HarfBuzz feature tags from the font for ligatures/kerning.
- **FONT-6 — round-trip test + example.** Render Latin/Cyrillic/Greek/Arabic/Hebrew/CJK; round-trip via `extract_text`; assert byte equality. Add `examples/create_pdf_with_custom_font.rs`.

5 sub-tasks instead of 6. Three days of implementation work removed.

### Revised PAINT sub-tasks

Plan v1 had PAINT-1..PAINT-7 (display list + Y-flip + gradients + shadows + transforms + clipping + opacity). Audit:

- Gradients (PAINT-3): `shading.rs` already has Type 2/3 builders → ~half a day to wire into PaintCmd, not 3 days.
- Transforms (PAINT-5): `cm` operator exists in ContentStreamBuilder → ~half a day to thread through.
- Clipping (PAINT-6): `W`/`W*`/`n` exist → ~half a day.
- Opacity (PAINT-7): `set_extgstate` exists → ~half a day.
- Shadows (PAINT-4): `SoftMask` builder is present but not connected to indirect refs (graphics_state.rs:256 TODO). Real ~2 days incl. multi-pass box blur approximation.
- Display-list walker (PAINT-1) + Y-flip (PAINT-2): the new code. ~2 days.

PAINT phase: ~10 days → **~6 days**.

## 3. HTML / CSS / Layout (no compression — confirmed greenfield)

Audit confirms there is **zero** write-side CSS, selector matching, cascade, computed styles, box tree, inline formatting, flex/grid, paginator, or page-break logic. The phases CSS / LAYOUT / PAGINATE start from scratch as planned.

Two starting-point items worth noting (do not change scope, but inform implementation):

- `src/pipeline/input_parsers/html.rs` (522 lines) — regex-based tag stripper for ~10 tags. Useful as a starting test fixture for the new HTML tokenizer (compare outputs); not a foundation to extend.
- `src/converters/html.rs` and `src/pipeline/converters/html.rs` are the *read* path (PDF → HTML). Different direction, different problem. `src/converters/html.rs` is already `#[deprecated(since = "0.2.0")]`.
- `src/writer/font_manager.rs:729+` `TextLayout::wrap_text()` — primitive greedy line breaker by width. Not reusable for inline formatting (no BiDi, no UAX#14, no script segmentation). Can be deleted once Phase LAYOUT lands.
- `src/writer/table_renderer.rs::TableLayout` — column-width / row-height calculation for the existing Markdown-to-PDF table path. Pattern for LAYOUT-7 table layout, not directly reusable.

Read-side `src/layout/` (XY-cut, clustering, multi-column detection) is a completely different problem (laying out *extracted* text from existing PDFs) and is not relevant.

### Compression flag — ship in v0.3.35 cheaply

`src/writer/pdf_writer.rs:60` has flate2 disabled by default. Enabling it produces ~3–5× smaller PDFs for typical HTML output. This is a one-character config change plus a regression-test pass; it ships in Phase API rather than as its own phase.

## 5. Open questions resolved by the audit

From the plan doc's open-questions section:

1. **`subsetter` crate vs hand-roll** → **use `subsetter`** (Typst uses it in production; we need real subsetting now since the existing `FontSubsetter` only tracks intent). Confirmed.
2. **TTF-only or CFF support** → **TTF only for v0.3.35**. The existing `TrueTypeFont` assumes the sfnt header layout; CFF/OpenType-CFF would need a separate parser. Document as a non-goal; defer to v0.3.36.
3. **`fontdb` on WASM** → confirmed problem (no filesystem). Behind `system-fonts` feature, default on for native bindings, default off for WASM. WASM users register fonts explicitly.

Still open from the original list (need user confirmation in week 1):

- `net` feature default — author still leaning on for bindings, off for Rust default.
- Font cache path (`~/.cache/pdf_oxide/fonts/` honouring XDG; `%LOCALAPPDATA%` on Windows).
- Host-absolute `<img src="/abs/path">` resolution.
- Deterministic font-family tie-break (lexicographic by face path — confirmed in FONT-4 description).
- Python API accepts `pathlib.Path`-like or str-only.

## 6. Execution order for this release

Updated, audit-informed:

1. **PREP-1** ✅ done — cargo-deny tightened.
2. **FONT-2** (subsetter) — independent of everything else; can start now. ~1 day.
3. **FONT-3** (PDF object emission + wiring) — depends on FONT-2 for subset bytes. ~1.5 days. *This is the unlocking step* — once it lands, every existing Base-14 callsite can switch to embedded fonts.
4. **FONT-4** (fontdb) — independent, can interleave. ~0.5 day.
5. **FONT-5** (rustybuzz) — needed before LAYOUT-3 inline formatting. ~1 day.
6. **FONT-6** (round-trip + example) — gates Phase FONT close. ~0.5 day.
7. **CSS** phase. Per plan.
8. **HTML** phase — start from regex-stripper test fixtures as comparison oracle. Per plan, ~3 days saved.
9. … remaining phases per plan.

Execution order remains "FONT first, everything else after" because every other phase has a transitive dependency on having an embedded-font emission path. The compression is in *how much new code* each phase needs, not in their ordering.
