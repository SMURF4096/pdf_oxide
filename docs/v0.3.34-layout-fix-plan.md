# v0.3.34 Layout-Fix Plan

**Status:** Blocking ship of v0.3.34. None of these fixes ship until all four phases land + regression validates.

> **TEMPORARY DOC.** Delete this file after TDD-7 validation passes and v0.3.34 is tagged. This is execution scaffolding, not permanent project documentation.

**Background.** Branch `release/v0.3.34` introduces commit `cb86499` (#319) which fixed the catastrophic multi-column interleaving bug (`accompaally`, `correlaanonymous`) by skipping the global row-aware re-sort when `is_multi_column_page` returns true. That fix is correct in spirit but the side effects of "skip row-aware sort" are bigger than realized. Regression analysis on 407 PDFs surfaced four distinct artifact classes that didn't exist in v0.3.33:

| Class | Example | Affected (subset) |
|---|---|---|
| Full sentence reversal | `NAVYTHEOFDEPARTMENT` | 1 PDF (L3YHC fax) |
| Hyphen drop across line break | `crosscollateralized` | ~30 PDFs |
| Adjacent cell concatenation | `instancesinstancesinstances` | ~15 PDFs |
| Distinct-region collision | `conceptcollaborative` | ~20 PDFs |

Root cause is structural: every threshold in the layout pipeline is a hardcoded absolute value (100 pt min column width, 15 spans per half, etc.), but the geometry that makes a page "multi-column" or a gap "between cells" is **proportional to the dominant body font** of the page. A PDF set in 6 pt or 24 pt produces wildly different absolute spacings for the same logical layout.

The plan is to make every layout threshold font-relative, gated on per-page font statistics computed once from the spans.

## Architecture: per-page font statistics

Every layout heuristic that currently uses a hardcoded constant becomes a multiple of font-derived measurements computed once per page.

```rust
/// Per-page font statistics, computed once from extracted spans and
/// reused by every downstream layout heuristic.
pub(crate) struct PageFontStats {
    /// Mode of font_size across spans, weighted by character count.
    /// "1 em" for the page.
    pub dominant_em: f32,

    /// Median baseline-to-baseline distance for adjacent same-column spans
    /// drawn in dominant_font. Approximates the page's set leading.
    pub dominant_line_height: f32,

    /// Average advance width of dominant-font glyphs. Used to convert
    /// "N characters wide" thresholds into points.
    pub dominant_char_width: f32,

    /// Identifier of the font used by ≥50 % of glyphs on the page
    /// (or the largest single font if no font hits 50 %). Used to
    /// detect "different region" boundaries — caption vs body.
    pub body_font_id: FontId,
}

impl PageFontStats {
    pub fn from_spans(spans: &[TextSpan]) -> Self { /* O(n) one pass */ }
}
```

Computation cost: O(n) over spans, one pass. Negligible vs the sort + XY-cut work that follows.

Plumbing: `PdfDocument::extract_spans` computes once, passes by reference into:
- `is_multi_column_page(spans, stats)`
- `XYCutStrategy::apply(spans, stats)`
- The span emitter that joins adjacent spans into output text
- `reorder_rowspan_labels`

No global state. Easy to test in isolation.

## Phase 1 — Fix L3YHC reversal: font-aware `is_multi_column_page`

**Files:** `src/document.rs:6616` (`is_multi_column_page`)

**Current:**
```rust
if width < 100.0 { return false; }       // hardcoded
let left_count.min(right_count) >= 15;   // hardcoded
```

**Replace with:**
```rust
if width < (40.0 * stats.dominant_char_width).max(80.0) { return false; }
// Density check: each side must average more than ~3 spans per em²
// of column area. Body text is typically 5-15 spans/em²; sparse
// fax fragments are 0.5-2.
let left_density = left_count as f32 / (left_area_em2);
if left_density < BODY_TEXT_DENSITY_THRESHOLD { return false; }
// Same for right
```

**Why this fixes L3YHC:** the fax has ~40 fragments scattered over the page. Each "half" has ≥15 fragments (current threshold passes). But the *density* of those fragments per em² is far below body-text density — the fragments are sparse one-word splatters, not stacked lines of body text. Density check rejects the page → row-aware sort runs → no reversal.

**Why this preserves CFR/academic:** body text is densely packed; density check passes easily.

**Tests:** `body_two_column.pdf` → multi_col true; `fax_scattered_fragments.pdf` → multi_col false; `body_single_column.pdf` → multi_col false.

**Risk:** Medium. Need to tune `BODY_TEXT_DENSITY_THRESHOLD`. Approach: measure densities on 20 representative PDFs from the corpus, pick threshold below the lowest body-text PDF and above the highest fax PDF.

## Phase 2 — Fix `instancesinstancesinstances`: vertical-gap-aware joining

**Files:** wherever spans are concatenated into output text — likely `src/document.rs::extract_text_with_options` and the converter pipeline.

**Current:** spans joined with empty string or single space when adjacent.

**Replace with:**
```rust
fn join_separator(prev: &TextSpan, curr: &TextSpan, stats: &PageFontStats) -> &str {
    let v_gap = curr.bbox.y - (prev.bbox.y + prev.bbox.height);
    if v_gap > 2.0 * stats.dominant_line_height { "\n\n" }     // paragraph
    else if v_gap > 0.7 * stats.dominant_line_height { "\n" }  // line
    else if h_gap > 0.5 * stats.dominant_char_width { " " }    // word
    else { "" }                                                 // glued
}
```

**Why this fixes `instancesinstancesinstances`:** three identical cells stacked vertically have v_gap > line_height between each → newline emitted between them.

**Why it doesn't break currently-correct cases:** two adjacent words on the same baseline have v_gap ≈ 0 → space emitted.

**Tests:** `table_simple_3x3.pdf` → cell texts separated by newlines, no concat.

**Risk:** Low-medium. Need to verify currently-passing tests still pass. Some PDFs may currently rely on the empty-join behavior.

## Phase 3 — Fix `crosscollateralized`: hyphen rejoin across line breaks

**Files:** likely the same span emitter as Phase 2.

**Current:** `cross-` (end of line) followed by `collateralized` (next line) joined with `\n` → `cross-\ncollateralized`. When `\n` is later normalized to space or empty, becomes `crosscollateralized` (hyphen lost via the `-\n` → `` collapse rule).

**Replace with:**
```rust
// Detect end-of-line hyphenation:
//   prev span ends with "-", same column as curr span,
//   v_gap ≈ line_height, same body font.
// → consume the "-" and emit empty separator (joined word).
// The hyphen is intentionally dropped, recovering "cross" + "collateralized" = "crosscollateralized"
// for compound and "auto" + "matic" = "automatic" for simple.
//
// We don't need a dictionary check — modern PDF authors always use end-of-line hyphens
// for word-break, never as part of the word itself (those use non-breaking hyphens U+2011
// or get authored in a single span).
```

Wait — looking at this more carefully: the bug isn't the hyphen-drop logic per se, it's that **column-context-aware hyphen detection currently misses cases where the rejoin happens across what was XY-cut into separate column ordering**. The fix is to do the rejoin pass *before* XY-cut splits the page, while the natural reading-order context is intact.

**Tests:** `body_two_column_hyphenated.pdf` → `cross-collateralized` rejoined correctly in both columns.

**Risk:** Medium. Hyphen handling is genuinely tricky — false positives reattach things like "self-driving" incorrectly. Mitigate with: only apply when the prev span's last char is `-` AND the immediately preceding character is alphabetic AND the next span's first char is alphabetic AND on next line of same column.

## Phase 4 — Fix `conceptcollaborative`: distinct-region detection

**Files:** `src/pipeline/reading_order/xycut.rs`

**Current:** XY-cut splits by horizontal density valleys; once split, all spans within a column block get concatenated regardless of font.

**Replace with:**
```rust
// Within each XY-cut leaf block, before joining spans, detect
// distinct regions by font-size discontinuity:
//   - If a span's font_size differs from its neighbours by >25 %,
//     it belongs to a separate region (caption, footnote, sidebar)
//     and gets a region-break separator (\n\n) before joining.
//   - This prevents body text from absorbing caption text into
//     mid-sentence positions.
fn region_break_required(prev: &TextSpan, curr: &TextSpan, stats: &PageFontStats) -> bool {
    let prev_em = prev.font_size;
    let curr_em = curr.font_size;
    let ratio = (curr_em / prev_em - 1.0).abs();
    ratio > 0.25 && curr.font_id != prev.font_id
}
```

**Why this fixes `conceptcollaborative`:** body text is 10-12 pt, the figure caption is 9 pt with a different font. Ratio > 25 % → region break inserted → caption never concatenates into body sentence.

**Tests:** `body_with_figure_caption.pdf` → caption text emitted as a separate paragraph, body text reads cleanly without caption interpolation.

**Risk:** Medium. Need to make sure section headers (which legitimately differ in size) don't get over-isolated to the point of breaking flow. Mitigate by requiring BOTH font_size difference AND font_id difference.

## TDD sequencing (ordering rewritten — tests before code)

The phases above describe the *target behaviour* per bug class. The execution order is strictly TDD: every test for every bug class is written and committed in the **red** state first; only then do we write code to make them green, one phase at a time. No fix lands until its test was already failing on `main`.

| TDD step | Work | Outcome | Blocked by |
|---|---|---|---|
| **TDD-1** | Build all 8 fixture PDFs via `PdfBuilder`. Write `tests/test_layout_fixtures.rs` with assertions for all 4 bug classes. **All tests must FAIL on current HEAD.** Commit the red bar. | Failing tests in CI prove every bug is captured. | — |
| **TDD-2** | Write unit tests for `PageFontStats::from_spans()` against synthetic span lists. Implement infrastructure to make those tests green. | `PageFontStats` ready, no behaviour change to existing code. | TDD-1 |
| **TDD-3** | Implement Phase 1 (font-aware `is_multi_column_page`). | `fax_scattered_fragments.pdf` test goes red→green. `body_two_column.pdf` and `body_single_column.pdf` stay green. | TDD-2 |
| **TDD-4** | Implement Phase 2 (vertical-gap-aware joining). | `table_simple_3x3.pdf` test goes red→green. | TDD-2 |
| **TDD-5** | Implement Phase 3 (hyphen rejoin). | `body_two_column_hyphenated.pdf` test goes red→green. | TDD-2 |
| **TDD-6** | Implement Phase 4 (distinct-region detection). | `body_with_figure_caption.pdf` test goes red→green. | TDD-2 |
| **TDD-7** | Run full regression (407 corpus + Charltsing + all 8 fixtures). | All green; tag `v0.3.34`. | TDD-3 ... TDD-6 |

Estimates per step: TDD-1 ≈ 4-5 h, TDD-2 ≈ 2-3 h, TDD-3..6 ≈ 3-4 h each, TDD-7 ≈ 1-2 h. **Total ~2-3 focused days**, with TDD-3..6 parallelizable after TDD-2.

Discipline: a fix-phase commit (TDD-3..6) is only valid if the corresponding fixture test is the **only** test that flipped from red to green. If any other test changes state (red→green elsewhere, or green→red), stop and reassess — that's a sign of unintended interaction.
